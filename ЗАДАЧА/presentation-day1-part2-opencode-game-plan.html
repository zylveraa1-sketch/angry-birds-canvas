Пара 2: как правильно работать с opencode

Цель: правильно запустить проектную работу, настроить workflow и сразу работать с GitHub/Git.

Итог пары: структура проекта, репозиторий, план и стартовый MVP.

Шаг 1: создать папку проекта

Вариант через терминал

mkdir my-game cd my-game
  • `mkdir` создаёт папку проекта.
  • `cd` переходит в неё.

Вариант через Файлы

  • Создать папку через файловый менеджер.
  • Открыть терминал в этой папке через контекстное меню.
  • Дальше запускать opencode уже из этой папки.

Ключевые slash-команды opencode

Работа с моделью

  • /models — выбрать модель.
  • /variants — выбрать effort.

Работа с чатами

  • /session — вернуться к прошлому диалогу.
  • /new — начать новый чат.

Важно про /session

Набор чатов зависит от папки, из которой запущен opencode. В двух разных папках будут два разных списка сессий.

Быстрые действия в терминале

Копирование и вставка

  • Чтобы скопировать: просто выделите текст.
  • Вставка: Ctrl + Shift + V.

Перенос файлов

  • Файл можно перетащить в терминал мышкой.
  • Путь вставится автоматически.

Шаг 2: сразу подготовить GitHub и Git

GitHub CLI

gh auth login
  • Авторизоваться перед началом работы.
  • После работы выйти из аккаунта:
gh auth logout

Локальный Git

git init
  • Создать локальный репозиторий.
  • Дальше коммитить изменения и пушить на сервер.
  • Или попросить opencode выполнить это.

Что вы должны сделать за эту пару

Обязательный минимум

  • Определить жанр и core-loop игры.
  • Разделить роли в паре.
  • Собрать MVP-план в 5-8 задач.
  • Создать репозиторий и стартовую структуру.

Чего избегать

  • Сразу идти в сложный 3D без MVP.
  • Писать “сделай всё” одним промптом.
  • Работать без проверок и коммитов.
  • Менять стек посреди пары без причины.

Только после setup переходим к игре

  • Сначала: папка проекта + opencode + GitHub CLI + git init.
  • Потом: планирование игры в Plan mode.
  • И только затем: реализация в Build mode.
Порядок пары: setup -> Plan mode -> выбор подхода -> Build mode -> проверка -> commit.

Шаблон промпта на старт проекта

Используйте каркас:

Цель: [какую игру делаем] Ограничения: [2D/3D, стек, дедлайн] MVP: [минимум фич] Критерий готовности: [как понять, что работает] Формат ответа: [план задач + порядок реализации]

Пример:

Сделай план MVP 2D-игры "Dodge Balls" на Canvas за 3 дня. Нужно: движение, коллизии, счёт, экран поражения. Дай 7 задач в правильной последовательности.

Два подхода: поэтапный и мгновенный

1) Поэтапный (лучше для 4-5)

  • Стартуем в Plan mode и разбиваем игру на этапы.
  • Каждый этап делаем в Build mode, потом проверяем руками.
  • Если что-то не так, просим исправить до готового состояния.
  • После готового этапа: просим opencode сделать git commit.
  • Подходит для более проработанных игр и большого числа ассетов.

2) Мгновенный (лучше для 3)

  • В Plan mode готовим one-shot prompt под MVP.
  • Запускаем в Build mode и получаем игру за один большой запрос.
  • Плюс: очень быстрый выход на MVP.
  • Минус: высокий техдолг, исправлять потом дольше и сложнее.
  • Первый git commit делаем сразу на первой MVP-версии.

Оба подхода стартуют с Plan mode; разница в том, насколько детально дробим реализацию.

Рабочий цикл в opencode

1

Сформулировать узкую задачу.

2

Дать контекст и ограничения.

3

Выполнить через tools.

4

Проверить запуск/лог/тест.

5

Сделать commit и перейти дальше.

Золотое правило: одна логическая задача = один commit.

Инструкция по ассетам

Как подготовить

  • Скачать ассеты заранее.
  • Загрузить в папку проекта.
  • Лучше сделать отдельную папку, например assets/.
  • Передать opencode путь к этой папке.
Рекомендуемый источник: kenney.nl (и для 2D, и для 3D).

Платформы и сопоставление

  • Дополнительные платформы: itch.io и sketchfab.com.
  • Какой ассет для какого элемента игры.
  • Пример: player, enemy, background, UI, sounds.
  • Если не указать соответствия, ИИ может подключить всё неверно.
Пример: \"Используй assets/player.png для игрока, assets/enemy.png для врага, assets/bg.png для фона, assets/hit.wav для столкновений\".

Общее задание для студентов

  • Сделать аналог любимой игры или другой выбранной игры.
  • Сдать минимально рабочую версию, в которую реально можно поиграть.
  • Ключевое условие: основная механика должна работать стабильно.

Критерии на 5 / 4 / 3

Оценка 5

  • Либо 10+ 3D моделей.
  • Либо 10+ 2D ассетов.
  • Плюс 3 анимации из тех же паков.

Оценка 4

  • Либо 5+ 3D моделей.
  • Либо 5+ 2D ассетов.
  • Плюс 2 анимации.

Оценка 3

  • Либо 1 3D модель.
  • Либо 1 2D ассет.
  • Фото/текстура — достаточно.

Git + GitHub + GitHub Pages: минимальный pipeline

1. Локально

  • init/add/commit
  • чистая история

2. GitHub

  • push в main
  • README + структура

3. Pages

  • публикация билда
  • проверка ссылки
На каждой контрольной точке и в конце пары: git push в GitHub. После этого можно попросить opencode дать актуальную ссылку на GitHub Pages.

Стек проекта и чего нужно избежать

Разрабатываем именно сайт

  • Не делаем desktop-игру (Python game, Godot script и т.д.).
  • Если 2D: HTML + CSS + JS + Canvas.
  • Если 3D: HTML + CSS + JS + Three.js.

Анти-паттерн

  • Не сваливать всё в один index.html на 2000 строк.
  • В монолите ИИ чаще ошибается при фиксе и доработках.
  • Дробите код на файлы, чтобы правки были точечными.
Минимальная архитектура: index.html + style.css + script.js (подключение и запуск) + game.js (ООП-класс игры).

Тестирование и отладка в браузере

Console

  • Открывайте DevTools и вкладку Console.
  • Красные ошибки копируйте и отправляйте ИИ.
  • Просите ИИ объяснить причину и дать точечный фикс.

Кэш и обновление страницы

  • Иногда изменения не видны из-за кэша.
  • Делайте принудительную перезагрузку: Ctrl + F5.
  • Потом снова проверяйте Console и поведение игры.

Если не получилось с первого раза

Алгоритм исправления

  • Зафиксировать точную ошибку.
  • Сузить задачу до одного дефекта.
  • Попросить план фикса в 2-3 шага.
  • После фикса сразу проверить запуск.

Чего не делать

  • Не переписывать весь проект.
  • Не менять 5 вещей одновременно.
  • Не коммитить непроверенный код.

Чекпоинт конца пары

  • Папка проекта создана и опенкод запущен из правильной директории.
  • GitHub CLI авторизован, локальный git репозиторий инициализирован.
  • Есть план MVP и стартовые задачи.
  • Есть первые коммиты, сделан git push, и есть актуальная ссылка на GitHub Pages.
