[25.05.2026 14:53] Леся: Почему веб-чат не закрывает проектную работу

Обычный чат полезен для обсуждений и объяснений, но почти не делает действий над проектом: не трогает файлы, не запускает команды, не проверяет сборку и тесты. Поэтому для учебной разработки нужен режим, где ИИ не только «думает», но и «действует».

Harness / opencode: тот же ИИ, но с инструментами

В рабочем режиме ИИ получает «руки»: Files, Shell, Git, Browser, Tests, MCP, Playwright, Deploy. Это означает: можно создать файл, изменить код, найти баг в проекте, запустить проверку и зафиксировать результат коммитом. Главное отличие от веб-чата не в «умнее/глупее», а в доступе к инструментам.

Лимиты opencode и рабочая стратегия

Ориентир: около 200 запросов на 5 часов для бесплатных моделей (например Big Pickle), окно плавающее. Расход зависит от модели и сложности запроса. Практика: не просить «сделай всё», а дробить задачи. Формула ритма: задача -> выполнение -> проверка -> commit.

7 ключевых tools, которые надо уметь

1) Создание файла.
Запрос: «Создай src/game/config.js с базовыми настройками canvas».
Проверка: файл появился, содержимое по задаче, запуск не сломан.

2) Редактирование файла.
Запрос: «Измени только spawnEnemy(): лимит скорости + комментарий; остальное не трогай».
Проверка: правка точечная, без побочек.

3) Поиск по файлам.
Запрос: «Найди, где используется playerHealth и почему уходит в минус».
Проверка: найдены все места и причинно-следственная цепочка.

4) Удаление файла.
Запрос: «Проверь old-ui.js; если не используется — удали и поправь импорты».
Проверка: нет битых импортов, сборка проходит.

5) Выполнение bash-команд.
Запрос: «Запусти npm run dev и npm run test, покажи что упало».
Проверка: есть реальный output, объяснение ошибки и следующий шаг.

6) Веб-поиск.
Запрос: «Найди актуальные подходы к collision в canvas и сравни 2 варианта».
Проверка: есть ссылки, плюсы/минусы, применимое решение.

7) Git-операции.
Запрос: «Сделай commit только по механике прыжка: feat: tune jump physics».
Проверка: один коммит = одна логическая задача, история читаемая.

Plan mode vs Build mode

Plan mode: анализ, планирование, декомпозиция; инструменты выполнения ограничены.
Build mode: реальная работа через tools, изменения в проекте, проверки.
Правильный поток: Plan mode -> согласовали план -> Build mode -> тесты/проверка -> commit.

Форматы данных для ИИ

TXT: быстро, но мало структуры.
MD: лучший универсальный формат для учебных задач (заголовки, списки, критерии).
JSON: удобен для конфигов, параметров и интеграций.

Пример MVP в MD:
Цель: 2D-игра за 3 дня.
Фичи: движение, счёт, экран поражения.
Критерий готовности: игра запускается локально.

Установка и резервные инструменты

Linux: ставим CLI opencode, проверяем командой opencode --help.
Windows: возможны CLI и GUI.
Cursor можно использовать как запасной GUI-вариант, но лимиты там часто нестабильны для учебной аудитории. Критичные задачи лучше делать в opencode.

Итог

Стабильный результат даёт не «магический промпт», а дисциплина процесса: маленькие задачи, явные проверки, чистые коммиты и работа в правильном режиме (Plan для структуры, Build для реализации).

Мини-чеклист перед каждым запросом к ИИ

1) Что именно меняем (файл, функция, модуль).
2) Какие ограничения нельзя нарушать.
3) Как проверим результат (команда, тест, поведение).
4) Что должно быть в коммите после шага.

Пример хорошего формата запроса

«В файле src/game/enemies.js измени только spawnEnemy(): добавь верхний предел скорости 6, оставь текущую формулу спавна, добавь короткий комментарий why. После правки запусти npm test и покажи только упавшие тесты, если они есть».

Такой формат снижает шум, уменьшает риск случайных изменений и экономит лимит запросов.

Типичные ошибки: большой размытый промпт, отсутствие промежуточных проверок, один «коммит на всё», слепое копирование без локальной валидации. Правильная привычка: сначала воспроизводимый локальный результат, затем фиксация в git.
[25.05.2026 14:53] Леся: Пара 2: как правильно работать с opencode

Цель пары: запустить проектную работу без хаоса. На выходе у команды должны быть: папка проекта, рабочий git/GitHub-пайплайн, понятный MVP-план и стартовая версия игры.

Шаг 1. Подготовить рабочую папку

Через терминал:
mkdir my-game
cd my-game

Или через файловый менеджер: создать папку, открыть в ней терминал и запускать opencode именно из этой директории.

Важно: контекст opencode привязан к папке. Если запускать из разных мест, будут разные наборы файлов и сессий.

Базовые slash-команды

/models — выбор модели.
/variants — уровень effort.
/session — вернуть прошлый диалог.
/new — начать новую сессию.

Шаг 2. Сразу настроить GitHub и Git

Авторизация GitHub CLI:
gh auth login

После занятия на общем ПК:
gh auth logout

Инициализация локального репозитория:
git init

Дальше цикл стандартный: add/commit/push. Это можно делать вручную или через opencode, но ответственность за качество коммитов остаётся у вас.

Что обязательно сделать за пару

1) Выбрать жанр и core-loop.
2) Разделить роли в паре.
3) Собрать MVP-план на 5-8 задач.
4) Создать репозиторий и стартовую структуру.

Чего избегать

- Старт с «сделай всё сразу».
- Попытка сложного 3D без MVP.
- Работа без запусков и проверок.
- Смена стека по ходу занятия без причины.

Последовательность работы

Сначала setup: папка + opencode + GitHub CLI + git init.
Потом Plan mode: определить архитектуру и порядок шагов.
Потом Build mode: реализовывать по задачам.
После каждого шага: проверка запуска/логов/тестов и отдельный commit.

Шаблон стартового промпта

Цель: [какую игру делаем]
Ограничения: [2D/3D, стек, срок]
MVP: [минимум фич]
Критерий готовности: [как проверяем]
Формат ответа: [план задач в порядке реализации]

Пример:
«Сделай MVP-план 2D-игры Dodge Balls на Canvas за 3 дня: движение, коллизии, счёт, экран поражения. Дай 7 задач по порядку».

Два рабочих подхода

1) Поэтапный (рекомендуется для оценок 4-5):
- Планируем в Plan mode.
- Реализуем этап в Build mode.
- Проверяем и исправляем до стабильного состояния.
- Делаем commit.
Плюс: выше качество и контроль, минус: медленнее.

2) Мгновенный one-shot (чаще для оценки 3):
- Готовим один большой промпт под MVP.
- Получаем быстрый первый результат.
- Потом долго вычищаем техдолг.
Плюс: быстро, минус: больше ошибок и хаоса в коде.

Рабочий цикл

1. Узкая задача.
2. Контекст и ограничения.
3. Выполнение tools.
4. Проверка фактом (запуск/тест/лог).
5. Commit и переход к следующей задаче.

Золотое правило: один commit = одна логическая единица.

Ассеты: как подготовить правильно

Скачайте ассеты заранее, сложите в assets/, заранее сопоставьте файлы с ролями: player, enemy, background, UI, sounds. Если не дать это соответствие явно, ИИ легко подключит ресурсы неправильно.

Оценивание (минимум)

На 5: 10+ моделей/ассетов + 3 анимации.
На 4: 5+ моделей/ассетов + 2 анимации.
На 3: минимум 1 ассет/модель (включая фото/текстуру).

Минимальный pipeline публикации

1) Локально: init/add/commit, чистая история.
2) GitHub: push в main, базовый README.
3) GitHub Pages: публикация и проверка ссылки.

Стек и ограничения

Проект должен быть веб-игрой:
2D: HTML/CSS/JS + Canvas.
3D: HTML/CSS/JS + Three.js.

Анти-паттерн: один огромный index.html на тысячи строк. В монолите сложнее дебажить, сложнее вносить правки и выше шанс поломок.

Итог

Если соблюдать порядок setup -> plan -> build -> verify -> commit, команда почти всегда успевает собрать рабочий MVP и корректно сдать результат в рамках пары.

Чекпоинт конца пары

К финалу у вас должно быть: рабочее локальное MVP, несколько осмысленных коммитов, push в GitHub main и проверяемая ссылка на Pages (если публикация уже включена). Если один из пунктов отсутствует, приоритет не «новые фичи», а закрытие этого долга.
